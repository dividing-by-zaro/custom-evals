# Implementation Plan: Custom LLM Evaluation MVP

**Branch**: `001-eval-mvp` | **Date**: 2026-03-05 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/001-eval-mvp/spec.md`

## Summary

Build a custom LLM evaluation framework with: (1) JSON-based eval items
with objective binary rubrics scored by an LLM judge, (2) a Python
runner supporting OpenAI, Anthropic, and local LLM providers, and
(3) a React dashboard with model leaderboard and per-domain
breakdowns.

## Technical Context

**Language/Version**: Python 3.11+
**Primary Dependencies**: openai, anthropic, pyyaml, pydantic (Python); React + Vite (dashboard)
**Storage**: JSON files (eval items in `evals/`, results in `results/`)
**Testing**: pytest
**Target Platform**: macOS / Linux (local development tool)
**Project Type**: CLI tool + web dashboard
**Performance Goals**: Process eval items sequentially with concurrent
judge calls; dashboard loads in <2s for up to 50 runs
**Constraints**: Must run via `uv run`; no database; credentials via
env vars only
**Scale/Scope**: MVP starts with 1 sample eval item, designed for
<100 items per eval set

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

| Principle | Status | Evidence |
| --------- | ------ | -------- |
| I. Objective Binary Scoring | PASS | All criteria score 0 or 1 via LLM judge with structured output. No partial credit. |
| II. Domain-Tagged Items | PASS | Every eval item has a required `domain` field. Results preserve domain for aggregation. |
| III. Provider-Agnostic Runner | PASS | Provider ABC with `complete()` method. OpenAI, Anthropic, and local implementations. Config externalized to `config.yaml` + env vars. |
| IV. Reproducible Results | PASS | Every run records: run_id, timestamp, provider, model, params, all rubric scores. Temperature defaults to 0. |
| V. Separation of Concerns | PASS | Eval data (JSON files), runner (Python script), dashboard (React app) are independent. Dashboard reads results JSON files without importing runner. |

No violations. No complexity tracking needed.

script (`run_eval.py`) at repo root for easy `uv run` invocation.

`dashboard/` that reads results JSON files directly.
